# Time Budget Rules

These rules are mandatory.

## Core distinction
This project distinguishes between:
- **Active training time** = time spent doing the work
- **Elapsed calendar time** = active time + scheduled breaks

Codex must show both.

## Daily schedule

| Day | Active time | Breaks | Elapsed time |
|---|---:|---:|---:|
| Monday | 2h (120m) | 10m | 2h10 (130m) |
| Tuesday | 2h (120m) | 10m | 2h10 (130m) |
| Wednesday | 4h (240m) | 30m | 4h30 (270m) |
| Thursday | 2h (120m) | 10m | 2h10 (130m) |
| Friday | Rest | 0 | Rest |
| Saturday | 4h (240m) | 30m | 4h30 (270m) |
| Sunday | 4h (240m) | 30m | 4h30 (270m) |

## Weekly totals
- **Active training time**: 1080 minutes = 18 hours
- **Elapsed calendar time**: 1200 minutes = 20 hours

## Break rule
Insert **one 10-minute break after each completed active hour**, except after the final active hour of the session.

Examples:
- 1h active = 1h elapsed
- 2h active = 2h10 elapsed
- 4h active = 4h30 elapsed

## Formatting rule for weekly plans
Every training day must be presented as a set of active-hour blocks.

### 2h day
Use:
- Hour 1 — 60m active
- Break — 10m
- Hour 2 — 60m active

### 4h day
Use:
- Hour 1 — 60m active
- Break — 10m
- Hour 2 — 60m active
- Break — 10m
- Hour 3 — 60m active
- Break — 10m
- Hour 4 — 60m active

## Internal sub-blocks
Within a 60-minute active hour, Codex may split time into sub-tasks if helpful, but the total must still add up to 60 active minutes.

Example:
- 35m: watch / read
- 25m: perform task / note output

## Overbooking rule
No day may exceed its active-time budget.
No day may hide extra work in vague "optional" tasks unless that optional work is explicitly marked as optional and not required for the week to remain valid.

## Friday rule
Friday is rest.
Codex may add a short note such as "rest / walk / no study", but no training tasks.

## Preferred planning rule
Use active time to budget work.
Use elapsed time only to communicate the real calendar footprint.

## Program realism rule
Early weeks should not over-allocate lab complexity just because active hours exist.
Difficulty must scale with skill, not with calendar availability.

## What the generated program should show
Every weekly file must make it visually obvious:
- how much work is planned,
- where the breaks sit,
- what the outputs are,
- and whether the plan is realistic.
