# Candidate Profile — Training-Relevant Version

## Candidate summary
The candidate is a **strong analytical self-learner** with advanced academic training and strong systems thinking, but is **new to operational offensive cybersecurity**.

This program should assume:
- high reasoning ability
- high tolerance for structured study
- willingness to produce notes and artifacts
- willingness to work through a fixed sequence
- beginner-to-early-intermediate status in practical cyber operations

## What the candidate already brings
- strong pattern recognition
- structured thinking
- discipline with complex material
- comfort with abstract models and systems
- beginner-to-early familiarity with:
  - Python
  - R
  - Linux basics
  - Git
  - network / security concepts at a conceptual level

## What the candidate does NOT yet have
- extensive real offensive operations experience
- deep hands-on enumeration and exploitation routine
- mature privilege escalation instinct
- a long history of lab-machine compromises
- OSCP-level speed

## Desired path
The candidate wants an **offensive path**.

That means the program should prioritize:
- enumeration
- web basics
- service analysis
- exploitation logic
- privilege escalation
- disciplined note-taking
- reporting
- OSCP-style progression

## Desired end state
At the end of the program, the candidate should be able to:
- independently enumerate lab targets
- identify common attack paths
- perform basic-to-intermediate exploitation on training targets
- perform Linux and Windows privilege escalation in training environments
- document findings in an OSCP-style format
- use a personal Obsidian vault as an execution and review system

## Starting resources (already chosen)
These are the starting foundation materials and must be integrated rather than replaced:
- Jason Dion — CompTIA Network+ (Udemy)
- Jason Dion — CompTIA Security+ (Udemy)

## Constraints
- The candidate wants a program that is explicit and finite.
- The candidate does not want vague "study X" advice.
- The candidate wants exact tasks.
- The candidate wants the output integrated into Obsidian.
- The candidate has a fixed weekly schedule with mandatory rest on Friday.
- Break time must be budgeted correctly and shown explicitly.

## Scheduling constraints
Active hours:
- Monday: 2h
- Tuesday: 2h
- Wednesday: 4h
- Thursday: 2h
- Friday: rest
- Saturday: 4h
- Sunday: 4h

Break rule:
- add one 10-minute break between each completed active hour
- 2h active session = 2h10 elapsed
- 4h active session = 4h30 elapsed

## Program design preference
The candidate is better served by:
- a structured progression
- explicit weekly outputs
- early hands-on practice
- repeated exposure to the same tools in slightly harder settings
- periodic consolidation
- late-stage mock simulations

The candidate is not well served by:
- passive-only study
- broad generic cyber overviews
- too many optional branches
- resource overload without sequencing
- reliance on ephemeral box names without fallback logic

## Planning assumption
Assume the candidate can work steadily through a 24-week program if the work is clearly specified and the vault supports execution.
